# B2B 수출 매칭 플랫폼 — 기획 · 설계서

> 한국 공급사와 검증된 해외 바이어를 연결하는 회원제 B2B 플랫폼
> 작성 목적: Claude Code 기반 개발 착수용 설계 기준 문서
> 기술 스택: **Next.js** (프론트엔드) · **Supabase** (백엔드/DB/Auth) · **GitHub** (저장소)

---

## 0. 문서 사용법

이 문서는 개발 착수의 "단일 기준(source of truth)"이다. 아직 **의도적으로 비워둔 항목**(12장 참조)은 MVP 이후로 미룬 것이며, 구조만 확장 가능하게 남겨둔다. 코딩 중 결정이 바뀌면 이 문서를 먼저 갱신한 뒤 코드에 반영한다.

### 0.1 개발 필수 규칙 (예외 없음)

- **하드코딩 절대 금지.** 다음은 반드시 외부화한다.
  - UI 문구 → i18n 언어팩(영어 기본 + 한국어, 확장 가능 구조). JSX/컴포넌트에 문자열 직접 삽입 금지.
  - 설정값·연결정보 → 환경변수(`.env`, Supabase URL/Key 등). 코드·저장소에 값 노출 금지.
  - 상수·enum·카테고리·역할·상태값 → 중앙 상수 모듈 또는 DB(`app_settings`, enum 테이블)에서 참조. 매직 넘버·매직 스트링 금지.
  - 라우트 경로, API 엔드포인트 → 상수로 정의 후 참조.
- **이모지 사용 금지.** 코드, 주석, 커밋 메시지, UI, 문서 어디에도 이모지를 넣지 않는다. 상태 표시는 텍스트 라벨 또는 아이콘 컴포넌트로 처리한다.

### 0.2 코딩 원칙 — 중복·오류 최소화 (Claude Code 착수 시 준수)

Claude Code로 개발할 때, 아래 원칙을 코드 작성 전·중·후에 지킨다. 목적은 중복 코드와 반복 오류를 구조적으로 줄이는 것이다.

**A. 문서 우선 (Doc-first)**
- 코드를 짜기 전에 이 문서(단일 기준)를 먼저 읽고, 만들려는 것이 문서의 어느 테이블·라우트·정책에 해당하는지 확인한다.
- 문서에 없는 새 결정이 필요하면, 코드부터 짜지 말고 **문서를 먼저 갱신**한 뒤 그 기준으로 코딩한다.
- 기능 착수 전, 이미 존재하는 유사 코드(컴포넌트·함수·타입·쿼리)를 먼저 검색한다. 같은 일을 하는 것이 있으면 새로 만들지 않고 **재사용·확장**한다.

**B. 단일 출처 원칙 (Single Source of Truth, 중복 방지)**
- 같은 지식을 두 곳에 쓰지 않는다(DRY). 상수·타입·검증 규칙·문구·쿼리는 **한 곳에 정의하고 import**한다.
- **타입 단일화**: DB 스키마에서 타입을 생성(예: Supabase 타입 생성)해 프런트·백엔드가 **같은 타입**을 공유한다. 손으로 타입을 중복 정의하지 않는다.
- **enum·상태값**: 0.1 규칙대로 중앙 상수/DB에서만 정의하고, 화면·검증·쿼리가 모두 그 하나를 참조한다.
- **권한·RLS**: 접근 규칙의 기준은 3.1 권한 매트릭스 하나다. 프런트 가드와 RLS가 같은 기준을 따르되, **최종 방어선은 항상 DB의 RLS**다(프런트 검사만으로 신뢰하지 않는다).
- **공통 컴포넌트**: 8장의 확인 다이얼로그·토스트·스켈레톤·비밀번호 토글·검색창 등은 **공통 컴포넌트 하나**로 만들어 전 화면에서 재사용한다(화면마다 복붙 금지).

**C. 오류를 구조로 막기**
- **TypeScript strict 모드** 사용. 타입으로 잡을 수 있는 오류는 런타임 전에 잡는다.
- **입력 검증 스키마**(예: Zod 등)를 한 곳에 정의해 폼·API 양쪽에서 재사용한다(검증 로직 중복·불일치 방지).
- **에러 처리 일관화**: 성공/실패 응답 형태, 오류 메시지 표기(8장 UX)와 로깅 방식을 공통 유틸로 통일한다.
- **린터·포매터**(ESLint·Prettier) 고정 설정으로 스타일·흔한 실수를 자동 차단. 커밋 전 검사.

**D. 작게, 검증하며 진행**
- 13장 로드맵의 Phase·단계 순서대로 **작은 단위로** 구현하고, 각 단계가 동작하는지 확인한 뒤 다음으로 넘어간다. 한 번에 많이 만들지 않는다.
- 각 기능은 **수직 슬라이스**(DB→API→UI 한 흐름)로 완성해 통합 오류를 조기에 발견한다.
- 핵심 로직(권한·중개·신원 보호)은 **테스트**로 고정해 이후 변경이 방어선을 깨지 않게 한다.

**E. 형상 관리 (GitHub)**
- 기능 단위로 **브랜치 → PR → 리뷰 → 병합**. 큰 덩어리 커밋 금지, 작은 커밋으로.
- 커밋·PR 설명은 "무엇을·왜"를 남긴다(이모지 없이, 0.1 규칙).
- 비밀키·환경변수는 절대 커밋하지 않는다(`.gitignore` 확인, 0.1 규칙).

> 요약: **문서에서 시작하고(문서 우선), 지식은 한 곳에만 두고(중복 방지), 타입·검증·RLS로 오류를 구조로 막고, 작게 나눠 검증하며 나아간다.**

## 1. 프로젝트 개요

### 1.1 핵심 자산과 철학
- 최대 강점은 10년간 쌓아온 **진성(眞性) 해외 바이어 네트워크**다. 알리바바 같은 오픈마켓과 달리 신뢰를 0에서 만들 필요가 없다.
- 해결하려는 진짜 문제는 "연결 부족"이 아니라 **"연결의 질 부족"** 이다 — 가짜·뜨내기 바이어와 진지하지 않은 공급사를 걸러내는 것.
- 일관된 원칙: **질을 먼저, 좁게 시작한다** (바이어·공급사 양쪽 모두 초대·승인제).

### 1.2 우선순위
- 절실한 쪽: **한국 공급사**
- 우선 타깃: **해외 바이어와의 B2B 거래**
- 성격: **B2C 아님** — 즉시 결제·구매 없음. 핵심 행동은 **문의 · 견적 요청**.

## 2. 기술 스택

| 계층 | 기술 | 역할 |
|---|---|---|
| 프론트엔드 | Next.js (App Router 권장) | UI, SSR/SSG, 라우팅 |
| 백엔드/DB | Supabase (PostgreSQL) | 데이터, Row Level Security(RLS) |
| 인증 | Supabase Auth | 이메일 기반 가입/로그인 |
| 저장/스토리지 | Supabase Storage | 제품 이미지, 인증서, 사업자등록증 |
| 실시간/알림 | Supabase Realtime | 토스트/알림박스 갱신 |
| 이메일 | 트랜잭션 메일 서비스(예: Resend/Postmark) | 가입·승인·문의·비밀번호 재설정 등 발송 |
| 저장소 | GitHub | 소스 코드 버전 관리 |
| 배포(권장) | Vercel | Next.js 배포 |

**국제화(i18n):** **영어(기본) + 한국어** 2개 언어로 시작한다.
- **영어**: 기본 언어. 해외 바이어 대상 전 화면의 기본값.
- **한국어**: 주로 **공급사·관리자**를 위한 언어(가입·제품등록·승인·중개 관리 화면 등 국내 사용자 흐름).
- 사용자별 언어는 `profiles.locale`에 저장하고, 로그인 시 해당 언어로 표시. 로그인 전(랜딩·목록)은 영어 기본 + 언어 전환 제공.
- 언어팩(예: `next-intl` 또는 `next-i18next`) 방식으로 구성해 추후 언어 추가가 가능하도록 한다. 모든 문구는 언어팩 키로 관리하며 코드에 하드코딩하지 않는다(0.1 규칙).

## 3. 4주체 구조와 관계

```
                [관리자 ADMIN]  ← 모든 흐름의 중심축 (문지기 · 중개자)
               /       |       \
      [공급사 SUPPLIER] |    [에이전트 AGENT]  ← 바이어 기능 겸함
                        |         |
                        |     [바이어 BUYER]
```

- **관리자(admin):** 승인·검증·중개·노출관리·에이전트 임명. 모든 소통이 이 사람을 거친다.
- **공급사(supplier):** 제품·미니홈페이지 운영. 바이어에게 직접 접근 불가.
- **에이전트(agent):** 하부 바이어 관리 + **본인도 바이어처럼 문의·견적 가능**(이중 역할). 관리자가 국가별·룰별로 **재량 선별·임명**.
- **바이어(buyer):** 제품 발견 + 문의·견적. 에이전트 추천링크로만 진입.

핵심 성격: **중앙집중형 중개 구조**. 관리자가 모든 관계의 통제권을 쥔다. (단, 미래 병목 대비 "직접 전달 스위치"를 미리 내장)

### 3.1 권한 · 가시성 매트릭스

| 대상 | 바이어 신원 | 거래 금액 | 제품 상세 | 직접 소통 |
|---|---|---|---|---|
| 관리자 | 전부 | 전부 | 전부 | 모두와 가능 |
| 에이전트 | **자기 하부만** | **가림** | 회원 열람 | 관리자 통해 |
| 공급사 | **가림** | 자기 거래만 | 자기 것 | 관리자 통해 |
| 바이어 | 본인만 | 본인만 | 회원 열람 | 관리자 통해 |

> **[중요]** 이 매트릭스는 Supabase **RLS 정책의 설계 기준**이다. 특히 "공급사에게 바이어 신원 가림", "에이전트에게 하부 바이어의 거래 금액 가림"은 반드시 RLS로 강제한다.

## 4. 3대 핵심 흐름

### 흐름 A — 진입 / 승인 (질을 좁게 거르는 관문)
```
공급사 :  이메일 가입 → 로그인 → 부가정보(사업자등록증·회사정보) → [관리자 승인] → 활동
에이전트:  관리자 초대링크 → 이메일 가입 → 활동
바이어 :  에이전트 추천링크 → 이메일 가입 → 로그인 → 부가정보 → 활동
```
- 회원가입은 **이메일 하나로만**, 로그인 후 부가정보 입력.
- 바이어 가입 방식은 관리자가 **자율가입 / 에이전트 초대 전용** 중 선택 가능.

### 흐름 B — 제품 노출 (진열대 채우기)
```
공급사 제품 등록 → [관리자 노출 승인] → 목록 노출(비회원도 목록·대표이미지까지 봄)
                                     → 상세페이지 (로그인 회원만)
                                     → 가격 (로그인 회원만, 공급사가 공개 선택 시)
```
- 초기 진입 전략: **믿을 수 있는 소수의 좋은 공급사부터 초대**해 진열대를 먼저 채운다(질 우선).

### 흐름 C — 문의 / 견적 중개 (플랫폼의 심장)
```
바이어/에이전트 [견적·문의 메시지]
      → [관리자 확인]
      → 공급사에 전달 (바이어 신원 가림, 내용만)
      → 공급사 회신 → 관리자 → 바이어

미래 스위치(설정값): 관리자 → 공급사 직접전달 ON/OFF  (기본 OFF)
```
- 공급사는 바이어에게 **직접 연락 불가**. 되물을 때도 관리자를 통한다.
- 병목 대비: `direct_forward_enabled` 같은 설정 플래그를 처음부터 스키마에 둔다(기본 false).

## 5. 데이터 모델 (Supabase 초안)

> 실제 컬럼은 개발 중 확정. 아래는 뼈대. 모든 테이블에 `id (uuid)`, `created_at`, `updated_at` 기본 포함 가정.

### 5.1 사용자 / 역할
```
profiles
  - id (uuid, = auth.users.id)
  - role: enum('admin','supplier','agent','buyer')
  - email
  - display_name
  - phone (nullable)
  - locale (default 'en')
  - status: enum('pending','approved','rejected','suspended','withdrawn')
  - memo (관리자 전용 내부 메모, nullable)   -- 회원 관리용
  - last_login_at (nullable)
  - created_by (nullable)            -- 관리자가 대신 생성한 경우

admin_audit_logs  -- 관리자의 회원/데이터 수정 이력 (책임 추적)
  - admin_id (FK profiles)
  - target_table, target_id
  - action: enum('create','update','approve','reject','suspend','delete','role_change')
  - before (jsonb), after (jsonb)
  - created_at

agents            -- 에이전트 부가정보
  - profile_id (FK profiles)
  - invited_by (FK profiles, admin)
  - country
  - referral_code           -- 하부 바이어 초대용 링크 코드

buyers            -- 바이어 부가정보
  - profile_id (FK profiles)
  - agent_id (FK agents, nullable)   -- 소속 에이전트(추천자)
  - company_name
  - country

suppliers         -- 공급사 부가정보
  - profile_id (FK profiles)
  - company_name
  - biz_reg_file (Storage 경로)      -- 사업자등록증
  - tier: enum('free','paid')        -- 유료/무료
  - verified: boolean                -- 인증마크
  - approved: boolean
```

### 5.2 공급사 미니홈페이지 / 제품
```
supplier_pages   -- 공급사 미니홈(회사소개·인증서)
  - supplier_id (FK)
  - about (rich text)
  - certificates (Storage 경로 배열/별도 테이블)

products
  - supplier_id (FK)
  - title, description
  - detail_body (rich text)          -- 상세 설명(개요·핵심특징·사양·취급주의 등 자유서술)
  - category_id (FK categories)      -- 초기엔 대분류만 사용 (아래 categories 참조)
  - price (nullable)
  - price_visible: boolean           -- 가격 공개 선택제 (기본 false → '가격 문의')
  -- ↓ buyKOREA(KOTRA) 제품상세 구조 참고한 B2B 무역 필드
  - moq (nullable)                   -- 최소주문수량 (예: 100 Pieces)
  - moq_unit (nullable)              -- 단위(Pieces, Box, Set 등)
  - lead_time_min, lead_time_max (nullable)  -- 리드타임 범위(일)
  - freight_type (nullable)          -- 운송 조건(협의 후 진행 등)
  - transport_modes (nullable)       -- 운송 수단
  - ship_from (nullable)             -- 출고지(예: Busan)
  - payment_terms (nullable)         -- 결제 조건
  - hs_code (nullable)               -- 품목 분류(수출용)
  - keywords (nullable)              -- 검색 키워드/태그
  -- (그 외 세부 필드는 12장 참조 — 점진 확장)
  - status: enum('draft','pending','listed','rejected')  -- 관리자 노출 승인
  - is_featured: boolean             -- 추천제품 우선노출

product_certifications  -- 제품 인증·수상 (buyKOREA의 Certification/Award 참고)
  - product_id (FK)
  - type: enum('certification','award')
  - name
  - file (Storage 경로, nullable)

categories        -- 제품 카테고리 (계층 구조 지원, 초기엔 대분류만 운용)
  - name
  - parent_id (FK categories, nullable)  -- 세부분류 확장 대비(초기엔 전부 null)
  - sort_order: int
  - is_active: boolean

product_media
  - product_id (FK)
  - type: enum('image','video_link','catalog_pdf')  -- 이미지·동영상링크·카탈로그PDF
  - url (Storage 또는 외부 동영상 링크)
  - is_primary: boolean              -- 대표이미지
  - sort_order: int
```
> **초기 카테고리 정책:** 제품이 적은 초기에는 **세부 분류 없이 넓은 대분류만** 운용한다(빈 세부 카테고리가 "물건 없음" 인상을 주는 것을 방지). `categories`는 `parent_id`로 계층 구조를 미리 지원하되, 공급사·제품이 충분히 늘면 **관리자가 세부 카테고리를 추가**한다. 카테고리 목록은 하드코딩하지 않고 DB에서 관리한다(0.1 규칙).

> **제품 등록 양식 참고 기준:** Commercial·Industrial 제품의 등록 항목·상세 레이아웃은 **KOTRA buyKOREA 제품상세 구조**를 벤치마크로 삼는다(다중 이미지+대표이미지, 인증/수상, MOQ·리드타임·운송·결제·출고지, 상세 서술, 카탈로그 PDF, 공유/QR). 참고 URL: buyKOREA 제품상세 페이지. 우리 상황에 맞게 가감한다.

### 5.4 알림 (앱 내 알림 + 이메일)
> 알림은 두 채널로 나간다. **앱 내 알림**(`notifications`, 토스트·알림박스)과 **이메일**(`email_outbox`). 중요 이벤트는 두 채널 모두, 사소한 것은 앱 내만.

```
wishlists
  - profile_id (FK)
  - product_id (FK)

notifications     -- 앱 내 알림(토스트/알림박스)
  - profile_id (FK)
  - type                              -- 유형별 분류(문의답변, 승인결과 등)
  - payload (jsonb)
  - read: boolean

email_outbox      -- 이메일 발송 큐 + 로그
  - to_email
  - profile_id (FK, nullable)         -- 대상 회원(비회원 대상도 있으므로 nullable)
  - template: enum('signup_verify','password_reset','supplier_approved',
                   'supplier_rejected','inquiry_received','inquiry_replied',
                   'rfq_response','agent_invite','event_reminder','generic')
  - locale                            -- en/ko (수신자 언어)
  - payload (jsonb)                   -- 템플릿 치환 값
  - status: enum('queued','sent','failed')
  - provider_message_id (nullable)    -- 발송 서비스 반환 ID
  - error (nullable)
  - sent_at (nullable)
```

**이메일 발송 정책**
- **발송 서비스**: 트랜잭션 메일 서비스(예: Resend, Postmark 등) 연동. 연결정보는 환경변수로만 관리(0.1 규칙).
- **트리거(핵심)**: 가입 인증, 비밀번호 재설정, 공급사 승인/반려, 문의·견적 도착(관리자에게)·회신 도착(바이어에게), RFQ 응답, 에이전트 초대, 행사 리마인더.
- **다국어**: 수신자 `locale`에 맞춰 템플릿 선택(en/ko). 템플릿 문구는 하드코딩하지 않고 언어팩/템플릿으로 관리.
- **중개 구조 준수**: 문의·견적 관련 메일도 **바이어 신원 보호 원칙**을 지킨다. 공급사에게 가는 메일에는 바이어 식별정보를 넣지 않는다(내용만).
- **발송 이력**: 모든 메일은 `email_outbox`에 기록(상태·실패사유·발송ID)해 재발송·감사에 대응.
- **수신 거부**: 마케팅성 메일은 수신거부(opt-out) 링크 필수. 트랜잭션 메일(가입·승인·보안)은 서비스 운영상 필수 고지.

### 5.7 게시판 — 바이어 참여형 제품요청글 (RFQ, 역방향 흐름)
> 바이어가 "이런 제품을 찾습니다"를 먼저 올리고, 공급사가 반응하는 **역방향 소싱 흐름**.
> 방향만 반대일 뿐 통제 구조는 기존과 동일: **관리자 승인 후 노출 + 관리자 중개 + 바이어 신원 보호**.

```
product_requests  -- 바이어가 작성하는 제품요청글
  - requester_id (FK profiles)       -- 작성자(바이어 또는 에이전트)
  - title, body (rich text)
  - category_id (FK categories, nullable)   -- products와 동일 분류 체계로 필터 일관성
  - target_country (nullable)
  - budget, qty (nullable)
  - status: enum('submitted','admin_review','listed','closed','rejected')  -- 관리자 승인 후 listed
  - is_pinned: boolean

product_request_responses  -- 공급사의 "응하겠다" 표시 (직접 답글/연락 아님)
  - request_id (FK)
  - supplier_id (FK)
  - message                          -- 관리자에게 전달되는 제안 내용
  - status: enum('submitted','forwarded_to_buyer','accepted','declined')
```

**열람 · 행동 권한 (제품요청글)**
- **상세 내용**: 비로그인자도 열람 가능 (진성 바이어 존재감을 외부에 드러내는 창구).
- **문의/요청/응답 버튼**: **로그인 사용자만** 사용 가능.
- **작성자(바이어) 개인정보**: **관리자를 제외한 누구도 볼 수 없다.** 로그인한 공급사·에이전트·다른 바이어 모두 실명·회사명·연락처·이메일 등 식별정보 접근 불가. 화면에는 마스킹된 신호(국가·업종·검증배지 정도)만 노출한다.
- 실제 연결은 `product_request_responses` → **관리자 중개**로만 이뤄진다. (3.1 권한 매트릭스와 동일 원칙)
- **[필수]** RLS 강제: `product_requests` 조회 시 `requester_id` 및 연결된 작성자 식별 컬럼은 관리자 role에게만 노출. 그 외 role에는 마스킹 뷰만 제공.

### 5.7-B 게시판 공통 — 에디터 · 첨부 · 인라인 미디어
> 위 게시판들(공지·FAQ·행사·EPC·RFQ 및 제품 상세 서술)의 본문은 공통 **리치 에디터**와 **첨부·미디어 기능**을 공유한다. 화면마다 따로 만들지 않고 **하나의 공통 에디터 컴포넌트**로 재사용한다(0.2 단일 출처 원칙).

**리치 텍스트 에디터 (rich text)**
- 서식: 제목/소제목, 굵게·기울임·밑줄, 목록(순서/비순서), 인용, 구분선, 링크, 표(선택).
- 붙여넣기 정리(불필요한 스타일 제거), 실행취소/재실행.
- 저장 형식은 안전한 구조화 포맷(예: 정제된 HTML 또는 JSON)으로 하고, **출력 시 새니타이즈(XSS 방지)** 를 반드시 거친다.
- 에디터 라이브러리는 개발 시 선택(예: TipTap 등). 문구·툴바 라벨은 언어팩으로 관리(0.1 규칙).

**첨부파일 · 미디어 (공통 테이블)**
```
board_attachments   -- 모든 게시판/제품 본문의 첨부·미디어 공통 관리
  - owner_type: enum('notice','faq','event','project','product_request','product')  -- 어느 게시물에 속하는지
  - owner_id                          -- 해당 게시물 id
  - kind: enum('image','video_file','video_link','file')
  - storage_path (nullable)           -- 업로드 파일(image/video_file/file)의 Supabase Storage 경로
  - external_url (nullable)           -- 동영상 링크(YouTube/Vimeo 등)일 때
  - file_name, mime_type, size_bytes (nullable)
  - thumbnail_path (nullable)         -- 이미지/영상 썸네일
  - inline: boolean                   -- true면 본문 안에 삽입, false면 하단 첨부목록
  - sort_order: int
```

**본문 내 인라인 삽입 · 재생**
- **이미지**: 본문 중간에 삽입해 그대로 렌더링(썸네일→클릭 시 원본 라이트박스). 대체텍스트(alt) 입력 지원(접근성).
- **동영상 파일**(업로드): 본문에 **플레이어로 삽입해 그 자리에서 재생**(HTML5 video). 썸네일·재생버튼 노출.
- **동영상 링크**(YouTube/Vimeo 등): URL을 붙이면 본문에 **임베드 플레이어로 자동 변환**해 인라인 재생. 원본 링크도 함께 보존.
- **일반 파일**(PDF·문서 등): 본문 삽입 대신 **첨부목록**에 파일명·용량·다운로드 버튼으로 표시(8장 다운로드 확인/피드백 적용).
- 삽입 미디어와 하단 첨부목록은 같은 `board_attachments`를 `inline` 플래그로 구분한다.

**업로드 규칙 · 안전**
- 허용 확장자·용량 상한은 하드코딩하지 않고 설정값(`app_settings` 또는 상수)으로 관리(0.1 규칙).
- 업로드는 8.3 로딩 표준을 따른다: 진행률 표시, 실패 시 사유·재시도. 대용량은 단계 안내.
- 파일 타입·용량 서버측 검증, Storage 접근 권한(RLS/정책)으로 무단 접근 차단. 외부 동영상 링크는 화이트리스트 도메인만 임베드 허용(임의 스크립트 삽입 방지).
- 작성 주체 권한은 각 게시판 규칙을 따른다(공지·FAQ·행사·EPC는 관리자만, RFQ는 바이어/에이전트, 제품은 공급사).

### 5.8 공유 링크 · 단축 URL · QR코드
> 제품 공유 링크와 회원가입 추천 링크는 **단축 URL + QR코드**를 함께 제공한다.
> (buyKOREA도 제품 상세에 공유·QR 기능을 제공 — 동일한 관행)

```
short_links       -- 단축 URL + QR 대상
  - slug (unique, 짧은 코드)          -- 예: /s/aZ3x9
  - target_type: enum('product','signup_referral','supplier_page','event','project','request','service')
  - target_id (nullable)             -- 대상 레코드
  - ref_code (nullable)              -- 추천링크일 때 에이전트/초대 코드 연결
  - created_by (FK profiles)
  - click_count: int                 -- 유입 추적
  - is_active: boolean
```
- **QR코드**는 단축 URL(`/s/<slug>`)을 인코딩해 생성. 이미지 저장 없이 렌더링 시 생성하거나 Storage에 캐싱(개발 시 선택).
- **회원가입 추천 링크**: 에이전트의 `referral_code`를 담은 `signup_referral` 단축링크 → QR로도 배포 가능. 스캔/클릭 시 `/auth/signup?ref=<code>`로 연결.
- **제품 공유 링크**: `product` 대상 단축링크 → QR 제공. 단, 비회원은 상세/가격 열람 제한 정책이 그대로 적용됨.
- **유입 추적**: `click_count`로 어떤 링크·에이전트가 효과적인지 측정(추후 정산 설계의 근거 데이터로도 활용 가능).

### 5.10 법적 페이지 · 회원 탈퇴 · 쿠키 동의
> **[주의]** 아래 약관·개인정보처리방침 초안은 **개발·구조용 자리 채움(placeholder)** 이다. 실제 오픈 전 **반드시 변호사 검토**를 받는다. 특히 해외(EU 등) 바이어의 개인정보를 다루고 국가 간 데이터 이전이 발생하므로 GDPR·ePrivacy 등 관할 규정 검토가 필요하다.

**법적 페이지 (관리자만 편집, 정적 콘텐츠)**
```
legal_documents
  - type: enum('terms','privacy','cookie_policy')
  - locale                           -- 언어별 버전(en/ko)
  - body (rich text)
  - version                          -- 개정 이력 추적
  - effective_date
  - is_current: boolean
```
- 초기값: **보편적 표준 초안**을 en/ko 각각 넣어둔다(이용약관, 개인정보처리방침, 쿠키 정책). 추후 법률 검토본으로 교체.
- 접근: 비회원 포함 누구나 열람. 푸터·가입 화면에 링크.

**회원 탈퇴 처리**
- 탈퇴 시 `profiles.status = 'withdrawn'`.
- 그 회원이 남긴 데이터(제품·문의·요청글)는 **삭제가 아니라 익명화 보존**을 기본으로 한다(거래 이력·통계 무결성 유지). 개인 식별정보만 파기.
- 단, 사용자가 개인정보 삭제를 명시적으로 요구하면(GDPR 삭제권) 관리자가 개인 식별정보를 파기하고 `admin_audit_logs`에 기록.

**쿠키 동의 (GDPR/ePrivacy 기준)**
```
cookie_consents   -- 동의 기록(감사 가능하도록 저장)
  - visitor_id (익명 식별자) 또는 profile_id (nullable)
  - necessary: boolean (항상 true)     -- 필수 쿠키(동의 불필요, 고지만)
  - functional: boolean
  - analytics: boolean
  - marketing: boolean
  - consented_at, policy_version
```
- **필수 쿠키만 예외**: 인증·세션·언어설정 등 서비스 동작에 반드시 필요한 쿠키는 동의 없이 사용하되 쿠키 정책에 고지한다.
- **비필수 쿠키(분석·마케팅·기능)**: 사용자가 능동적으로 **동의(opt-in)하기 전에는 설정·실행 금지.** 사전 동의 전 스크립트 로드 차단.
- **배너 요건**: 첫 방문 시 표시, "모두 수락"과 **"모두 거부"를 동등하게 첫 화면에 배치**(다크패턴 금지), 카테고리별 선택 가능, 쿠키 정책 링크 포함, **언제든 철회 가능**.
- **동의 기록 보관**: 누가·언제·어떤 목적에 동의했는지 `cookie_consents`에 저장(감사 대응).
- 구현은 자체 배너 또는 CMP(동의관리 플랫폼) 연동 중 선택(개발 시 결정).

> **RLS 필수 지점:** `inquiries`/`inquiry_messages`에서 공급사 조회 시 `requester_id` 및 바이어 식별 정보는 **절대 노출하지 않는다**(뷰 또는 정책으로 마스킹). 에이전트는 자기 `agent_id`에 속한 바이어만, 그중 거래금액 컬럼은 제외하고 조회.

## 6. 랜딩페이지 구성

### 6.1 원칙
- **주인공은 제품이다.** 쇼핑몰·이커머스처럼 제품이 전면에 깔린 활기찬 화면. 텍스트 위주 "바이어 소개서"로 흐르지 않는다.
- 공급사 설득은 별도 코너가 아니라 **활기찬 제품 화면에 자연스럽게 스며든 신호**로 처리한다(번화한 시장 풍경 자체가 새 상인을 부르는 원리).
- 회원제 원칙과 양립: **구경은 자유, 거래는 회원.** 로그인 벽은 흥미가 생긴 뒤 결정적 지점(가격·거래조건·문의)에 세운다.

### 6.2 방문자별 첫 화면
- **비회원(처음/둘러보기)**: 제품 중심 랜딩을 자유롭게 스크롤. 목록·대표이미지·제목·카테고리·공급사명까지 노출.
- **로그인 회원**: 역할별 대시보드(7장) 중심으로 진입. 랜딩 대신 발견+진행상황 화면.

### 6.3 섹션 구성 (위→아래)
1. **히어로**: 대표 추천 제품/배너 캐러셀(유료 공급사 우선노출 위치). 강한 비주얼로 활기 전달.
2. **익명 활동 신호**: "이번 주 N개국 바이어가 견적 요청" 등. **바이어 신원은 가림**, 존재감만. → 공급사 설득의 증거물.
3. **추천·신규 제품 그리드**: 대표이미지+제목+카테고리 중심의 쇼핑몰형 진열. 무한스크롤/더보기.
4. **카테고리 진입**: Commercial / Industrial 등 대분류 바로가기(초기엔 대분류만).
5. **정보·서비스 하이라이트**: Event(다가오는 행사), Service(예: 태국 식약청 등록 지원), Notice 요약.
6. **공급사 입점 안내(작은 진입점)**: "한국 공급사 입점 문의" CTA. 전면이 아니라 한켠에 배치.
7. **푸터**: 회사정보, 약관·개인정보·쿠키정책 링크, 언어 전환.

### 6.4 비회원 열람 경계 (로그인 게이트)
제품을 클릭하면 상세로 진입하되 **일부까지만 노출**하고 결정적 지점에서 로그인 벽을 세운다.

| 구분 | 비회원에게 보임 | 로그인해야 열림 |
|---|---|---|
| 제품명·대표이미지·추가이미지 | O | — |
| 제품 소개(설명 텍스트) | **전부 O** (흥미 유발) | — |
| 카테고리·공급사명 | O | — |
| **가격** | X (절대 비공개) | O(회원 & 공개 제품만) |
| **거래조건**(MOQ·리드타임·결제·운송·출고지) | X | O |
| **문의·견적 버튼** | X | O |
| 상세페이지 하단 전체상세·카탈로그 | X | O |

- 벽에 막히는 지점에는 "가격·거래조건을 보려면 로그인/가입" 안내를 노출해 **가입 전환**을 유도(흥미가 생긴 직후라 전환율↑).
- 이 경계는 5.2 제품/가격 정책 및 10장 라우트(`/products/[id]`)와 일치.

### 6.5 광고배너 (관리자 직접 제작)
관리자가 **이미지와 문구를 직접 작성**해 랜딩에 노출하는 배너. 주로 유료 공급사 홍보용.
```
ad_banners
  - title                            -- 관리자 식별용(사용자에게 안 보임 가능)
  - image (Storage 경로)             -- 관리자 업로드
  - headline, subtext (nullable)     -- 관리자 작성 문구
  - link_url                         -- 클릭 시 이동(제품/공급사홈/외부 등)
  - placement: enum('hero','mid','sidebar')  -- 랜딩 내 노출 위치
  - supplier_id (FK, nullable)       -- 연계된 유료 공급사(선택)
  - sort_order: int                  -- 캐러셀 내 순서
  - start_at, end_at (nullable)      -- 게재 기간(예약 노출)
  - is_active: boolean
  - click_count, impression_count: int  -- 성과 측정
```
- 랜딩 6.3의 "히어로 캐러셀"과 "정보·서비스" 사이 배너 슬롯이 이 데이터로 채워진다.
- 문구·이미지는 하드코딩하지 않고 전부 이 테이블에서 관리(0.1 규칙). 다국어 문구가 필요하면 headline/subtext를 언어별로 확장.
- 게재 기간(start_at/end_at)으로 **예약 노출·자동 종료** 지원.

### 6.6 팝업 (관리자 설정 노출)
관리자가 만들어 조건에 따라 띄우는 팝업(공지·이벤트·프로모션 등).
```
popups
  - title
  - content_type: enum('image','rich_text','image_with_text')
  - image (Storage 경로, nullable)
  - body (rich text, nullable)
  - link_url (nullable)              -- 클릭 시 이동
  - target: enum('all','guest','buyer','supplier','agent')  -- 노출 대상
  - pages (nullable)                 -- 노출 페이지(예: 랜딩만 / 전체)
  - start_at, end_at (nullable)      -- 노출 기간
  - priority: int                    -- 동시 다수일 때 우선순위
  - dismiss_option: enum('close_only','today_off','week_off')  -- "오늘 그만보기" 등
  - is_active: boolean
  - view_count, click_count: int
```
- **노출 대상 지정**: 전체 / 비회원만 / 특정 역할만 선택 가능(예: 공급사에게만 입점 프로모션 팝업).
- **"오늘 하루 보지 않기"** 등 재노출 억제 옵션. 이 선택 상태는 쿠키 또는 서버(로그인 사용자) 중 정책에 맞게 저장한다. 단, 이때 쓰는 쿠키가 기능 쿠키에 해당하면 5.10 쿠키 동의 정책을 따른다(동의 전에는 기억 기능이 동작하지 않을 수 있으며, 그 경우 팝업은 매번 뜨되 닫기는 가능).
- 기간·우선순위로 여러 팝업을 순차/조건 노출.

## 7. 주체별 대시보드

### 바이어 대시보드 (로그인 동기 = 발견 + 추적, 둘 다)
- 상단: **내 문의·견적 진행 상황** (대기 / 회신 도착 / 완료)
- 중단: **추천·신규 제품**, 익명 활동 신호("이번 주 N개국 바이어 활동")
- 하단: 관심제품(위시리스트), 최근 본 제품, **내 제품요청글(RFQ) 상태**
- 핵심 버튼: 구매(X) → **견적요청 · 문의 · 제품요청글 작성(O)**

### 에이전트 대시보드 (바이어 대시보드 + 관리 레이어)
- 바이어 대시보드 기능 전체 포함
- 추가: **하부 바이어 리스트 + 활동상태** (기본정보 + 활동상태까지만 / **거래금액 가림**)

### 공급사 대시보드
- 상단: 내 제품 상태 (노출중 / 승인대기 / 반려)
- 중단: 도착한 문의·견적 (익명 바이어 기준) + 회신 대기, **바이어 제품요청글(RFQ) 열람·응답**
- 하단: 미니홈페이지 관리(회사소개·제품·인증서), 유·무료 등급 상태
- 익명 활동 신호로 "진성 바이어 존재감" 체감

### 관리자 대시보드 (운영 관제탑)
- 최우선: **승인 대기 큐** (공급사 가입 / 제품 노출 / 바이어 / **제품요청글 노출**)
- 다음: **중개 대기 큐** (신규 문의→공급사 전달 / 공급사 회신→바이어 전달 / **공급사 RFQ 응답→바이어 전달**)
- 관리: 에이전트 임명(재량 선택제), **랜딩 광고배너·팝업 제작/노출 관리**, 추천제품 편성, **회원 전권 관리**, **공지사항·FAQ·행사·서비스·EPC 게시·관리**, **메인 메뉴 편집**, 시스템 설정

#### 관리자 회원 관리 기능 (전권)
관리자는 모든 회원의 정보를 조회·수정·통제할 수 있다.
- **회원 목록**: 역할/상태/국가/가입일/최근로그인 필터·정렬·검색, 페이지네이션, CSV 내보내기.
- **회원 상세**: 프로필 전체(부가정보 포함), 활동 이력, 소속 관계(에이전트↔하부 바이어, 공급사↔제품).
- **수정 권한**: 회원 정보 편집, **역할 변경**(예: 바이어→에이전트 승격), 상태 변경(승인/반려/정지/탈퇴처리), 공급사 등급(무료↔유료)·인증마크 부여, 내부 메모 기록.
- **대리 조치**: 관리자가 회원을 대신 생성·초대, 비밀번호 재설정 링크 발송.
- **책임 추적**: 모든 변경은 `admin_audit_logs`에 before/after로 기록(누가·언제·무엇을).
- **[필수]** 이 전권은 **admin role에게만** RLS로 허용. 다른 role은 자기 데이터만.

## 8. 사용자 안내 · 상호작용 UX 표준 (전 화면 공통)

모든 화면·컴포넌트에 일관되게 적용하는 UX 규칙. 여기 정의된 패턴은 특정 화면이 아니라 **전역 공통**이며, 문구는 하드코딩하지 않고 언어팩(en/ko)으로 관리한다(0.1 규칙). 이모지는 사용하지 않고 텍스트·아이콘 컴포넌트로 표현한다.

### 8.1 행동 버튼의 2단 안내 (확인 → 실행 → 결과)
사용자의 주요 행동은 **확인 → 처리중 → 결과 피드백**의 3박자를 따른다.

| 행동 | 1) 확인(다이얼로그) | 2) 처리중 | 3) 결과 |
|---|---|---|---|
| 저장 | "저장하시겠습니까?" (일반 저장은 생략 가능) | 버튼 로딩 스피너 | 토스트 "저장되었습니다" |
| 삭제 | "삭제하시겠습니까? 되돌릴 수 없습니다." (파괴적: 강조색·재확인) | 버튼 비활성+스피너 | 토스트 "삭제되었습니다" |
| 제출(문의·견적·요청글) | "제출하시겠습니까?" | 버튼 로딩 | 토스트 "제출되었습니다. 관리자 검토 후 전달됩니다." |
| 승인/반려(관리자) | "승인(반려)하시겠습니까?" | 버튼 로딩 | 토스트 + 큐에서 항목 이동 |
| 발송(이메일·초대) | "발송하시겠습니까?" | 로딩 | 토스트 "발송되었습니다" |
| 로그아웃/탈퇴 | 재확인 필수(탈퇴는 영향 고지) | 로딩 | 결과 화면/토스트 |

- **확인 다이얼로그 기준**: 되돌릴 수 없거나 영향이 큰 행동(삭제·탈퇴·발송·승인/반려·상태변경)은 **반드시 확인** 단계를 둔다. 단순·가역 행동(임시저장, 위시리스트 토글)은 확인 생략하고 결과 피드백만.
- **파괴적 행동**: 삭제·탈퇴 등은 확인 버튼을 강조색으로, 필요 시 대상명을 다시 보여주거나 재입력 요구.
- **결과 피드백**: 성공은 토스트(자동 소멸), 실패는 사유가 담긴 인라인/토스트 오류 메시지. 성공·실패 모두 반드시 피드백을 준다(무반응 금지).

### 8.2 비밀번호 필드
- 비밀번호 입력란은 **표시/숨김 토글(눈 아이콘)** 을 기본 제공. 기본은 숨김.
- 아이콘에는 접근성 라벨(예: "비밀번호 표시/숨김") 부여, 키보드 포커스 가능.
- 회원가입·변경 시 **강도 표시**와 규칙 안내(길이·조합)를 인라인으로. 확인용 재입력 필드는 불일치 시 즉시 인라인 안내.

### 8.3 로딩 · 처리중 상태 (멈춘 것으로 오인 방지)
사용자가 "멈췄다"고 오인하지 않도록, 즉시 반응하지 않는 모든 지점에 로딩 상태를 표시한다.

- **버튼 단위**: 클릭 즉시 버튼을 로딩 상태(스피너+비활성)로 바꿔 **중복 클릭 차단**.
- **영역 단위**: 목록·대시보드·상세 로딩 시 **스켈레톤 UI**(빈 골격) 표시. 스피너보다 체감 대기가 짧다.
- **페이지 전환**: 라우트 전환·데이터 fetch 시 상단 진행 인디케이터 또는 스켈레톤.
- **긴 작업**(이미지 업로드·대량 처리): 진행률 또는 단계 안내. 3초 이상 예상되면 "처리 중입니다" 문구 노출.
- **빈 상태(empty state)**: 데이터가 없을 때 빈 화면 대신 안내 문구+행동 유도(예: "아직 등록한 제품이 없습니다. 제품 등록하기").
- **오류 상태**: 실패 시 원인과 다음 행동(재시도 버튼 등)을 함께 제시.

### 8.4 폼 입력 안내
- **인라인 검증**: 제출 후가 아니라 입력 중/포커스 이탈 시 즉시 검증 안내.
- **필수 항목 표시**, 형식 예시(placeholder), 오류는 해당 필드 바로 아래 구체 문구로.
- **제출 방지**: 필수 미입력 시 제출 버튼 비활성 또는 첫 오류 필드로 포커스 이동.
- 저장되지 않은 변경이 있는 상태에서 페이지 이탈 시 "저장하지 않은 변경이 있습니다" 확인.

> 위 문구는 모두 예시이며 언어팩 키로 관리한다. 컴포넌트(확인 다이얼로그, 토스트, 스켈레톤, 비밀번호 토글)는 공통 컴포넌트로 만들어 전 화면에서 재사용한다.

### 8.5 실시간 검색 (필요한 화면 공통)
검색이 필요한 화면은 **입력하는 즉시 결과가 갱신되는 실시간 검색**(as-you-type)을 기본으로 한다. 제출 버튼을 눌러야만 결과가 나오는 방식이 아니다.

**적용 대상**
- 제품 목록/카테고리(`/products`) — 제품명·키워드·공급사명
- 제품요청글(RFQ, `/requests`), EPC(`/epc`), 행사(`/events`), 서비스(`/services`), 공지·FAQ
- 공급사 미니홈 내 제품
- 관리자 화면 — 회원 목록, 승인·중개 큐, 게시물 관리(가장 실시간성이 중요)

**동작 규칙**
- **디바운스**: 입력이 멈춘 뒤 약 200~300ms 후 질의(매 타건마다 요청 금지, 서버 부하·깜빡임 방지).
- **로딩 표시**: 검색 입력창 옆 인라인 스피너 또는 결과 영역 스켈레톤(8.3). 결과 교체 시 화면 점프 최소화.
- **결과 없음**: 빈 화면이 아니라 "검색 결과가 없습니다" + 대안(필터 초기화·전체 보기) 안내(8.3 빈 상태와 동일 원칙).
- **입력 보조**: 지우기(x) 버튼, 최근 검색어·추천어(선택), 키보드 접근성.
- **필터 연동**: 카테고리·국가·상태 등 필터와 검색어가 함께 적용되고, 필터 변경도 실시간 반영.
- **권한·게이트 준수**: 검색 결과에도 열람 경계가 그대로 적용된다. 비회원 결과에는 가격·거래조건이 나오지 않고(6.4), 바이어 개인정보는 검색으로도 노출되지 않는다(3.1). 각 role은 RLS가 허용한 범위 안에서만 검색된다.

**구현 방식(개발 시 선택)**
- 초기·소규모: Supabase(Postgres) `ilike`/트라이그램(`pg_trgm`) 또는 **전문검색(`tsvector` + GIN 인덱스)** 로 충분. 실시간 갱신은 프런트 디바운스 + 서버 질의로 구현.
- 대규모·오타보정·랭킹 고도화가 필요해지면 전용 검색엔진(예: Postgres 확장 또는 외부 검색 서비스) 도입 검토(11장/보류와 연계). 검색어 로그를 남기면 인기 검색어·개선 근거로 활용 가능.
- 검색 대상 필드·가중치는 하드코딩하지 않고 설정/상수로 관리(0.1 규칙).

## 9. 메인 메뉴 · 정보 구조 (IA)

메인 메뉴는 겉보기엔 한 줄이지만 속은 **네 가지 서로 다른 화면 유형**으로 나뉜다. 성격이 다른 만큼 **묶음으로 그룹화**하여 사용자 혼란을 줄인다.

> **[중요] 전체 메뉴 편집 권한:** 아래 표의 메뉴명은 **초기 기본값일 뿐**이다. **모든 메뉴에 대해 관리자가 표시명(라벨), 연결 링크(경로), 순서, 노출 여부, 그룹을 자유롭게 변경**할 수 있다. 특정 메뉴만이 아니라 **예외 없이 전부** 해당한다. 하드코딩 금지, `menu_items` 데이터로 관리(0.1 규칙).

| 그룹 | 메뉴(초기 기본명) | 화면 유형 | 작성/노출 주체 | 비고 |
|---|---|---|---|---|
| **제품** | Commercial | 제품 분류함 | 공급사 등록 → 관리자 승인 | 화장품·건강기능식품 등 잡화 |
| | Industrial | 제품 분류함 | 공급사 등록 → 관리자 승인 | 공업용 제품 |
| **프로젝트 & 요청** | EPC | 프로젝트 게시판 | **관리자 작성** | 발전소·건설·공장건설 등 |
| | Sourcing Requests | 요청 게시판(RFQ) | 바이어 작성 → 관리자 승인 | 기존 "Buy & Sell". 추천 표시명: **Sourcing Requests** 또는 **Buyer Requests** |
| **정보 & 서비스** | Event | 행사 게시판 | 관리자 작성 | 전시·행사 일정/소개 |
| | Service | 서비스 목록 | **관리자가 서비스 추가** | 예: 태국 식약청 등록 지원 |
| | Notice | 공지 게시판 | 관리자 작성 | 공지사항 |
| **(보류)** | Networking | 미정 | — | 지금은 노출 안 함(`is_active=false`). 추후 관리자가 활성화 |

> 위 "메뉴명"은 전부 관리자가 바꿀 수 있는 초기값이다. 예: Commercial→"Consumer Goods", Sourcing Requests→"Buyer Requests" 등 자유 변경. 연결 링크도 관리자가 재지정 가능.

- **제품 분류함**(Commercial, Industrial): 위 `categories`의 최상위 대분류에 해당. 클릭 시 해당 분류의 제품 목록. 초기엔 세부분류 없이 대분류만.
- **EPC**: 관리자가 올리는 프로젝트 게시판. 발전소·건설·공장건설 등 대형 프로젝트로 Event와 필드가 달라 **별도 `projects` 테이블**로 관리(5.6).
- **Sourcing Requests (구 Buy & Sell)**: 5.7의 `product_requests`(RFQ). 메뉴 표시명은 관리자가 변경 가능(추천: Sourcing Requests / Buyer Requests).
- **Service**: 관리자가 운영하는 **서비스 카탈로그**. 서비스가 늘어날 수 있으므로 전용 페이지가 아니라 데이터로 관리(아래 `services` 테이블).
- **Networking**: 현재 보류. 메뉴 `is_active=false`로 숨겨두고, 성격이 정해지면 관리자가 활성화.

```
services          -- 관리자가 구성하는 기타 서비스 카탈로그
  - title, summary, body (rich text)
  - cover_image (nullable)
  - status: enum('draft','published')
  - sort_order: int

menu_items        -- 메인 메뉴 자체를 데이터로 관리(관리자가 전 항목 편집)
  - label_en, label_ko               -- 표시명(언어별). 관리자가 자유 변경
  - group                            -- 묶음: 'product' | 'project_request' | 'info_service'
  - route                            -- 연결 링크(경로). 관리자가 재지정 가능
  - sort_order: int                  -- 순서
  - is_active: boolean               -- 노출 여부(Networking 등 숨김/활성화)
  - is_system: boolean               -- 핵심 메뉴 표시(삭제 방지용, 이름·링크·노출은 여전히 변경 가능)
```

## 11. 주요 정책 요약

| 정책 | 결정 |
|---|---|
| 가입 | 이메일 하나로 가입 → 로그인 후 부가정보 |
| 가입 모드 | 기본 초대·승인제 / 관리자가 열린광장 전환 가능(`registration_mode`) |
| 제품 상세 열람 | **로그인 회원만** |
| 카테고리 | 초기엔 **대분류만** 운용(세부분류 X). DB 관리·계층구조 지원, 제품 증가 시 관리자가 세부 추가 |
| 가격 표시 | 공급사 **선택제**(공개 / '가격 문의') · **비회원은 절대 못 봄** |
| 소통 | 관리자 중개 필수, 공급사↔바이어 직접 연락 불가, **바이어 신원 가림** |
| 노출 차등 | 유료: 인증마크 · 랜딩 광고배너 · 추천제품 우선노출 |
| 광고배너 | 관리자가 **이미지·문구 직접 작성**, 위치·게재기간·성과측정. 유료 공급사 홍보용(`ad_banners`) |
| 팝업 | 관리자가 제작·노출. **대상(전체/역할별)·기간·우선순위·재노출억제** 설정(`popups`) |
| 언어 | **영어(기본) + 한국어**(공급사·관리자용). `profiles.locale` 저장, 언어팩 구조로 확장 대비 |
| 알림 | 앱 내(토스트·알림박스) + **이메일**(가입·승인·문의·회신·비밀번호 재설정 등). 공급사향 메일도 바이어 신원 가림, 발송 이력 `email_outbox` 기록 |
| 바이어 노출 | 개인 신원은 감추고 **"검증된 바이어 활동 중"** 존재감만 신호화 |
| 게시판(공지형) | 공지사항·FAQ·행사일정은 **관리자만 작성**, 회원·비회원 읽기 전용 |
| 게시판(요청형/RFQ) | 바이어 작성 → **관리자 승인 후 노출**. 상세는 비로그인도 열람, **응답버튼은 로그인만**. 바이어 개인정보는 **관리자 외 누구도 열람 불가**(로그인 여부 무관), 응답은 관리자 중개 |
| 게시판 에디터·미디어 | 공통 리치 에디터 + 첨부(`board_attachments`). 이미지·동영상은 본문 인라인 삽입·재생, 링크는 임베드, 파일은 첨부목록. 새니타이즈·용량제한·도메인 화이트리스트(5.7-B) |
| 회원 관리 | 관리자가 **회원 전체를 조회·수정·역할변경·상태변경·등급부여** 가능(전권). 모든 변경은 `admin_audit_logs`에 기록 |
| 공유·추천 | 제품 공유링크·회원가입 추천링크에 **단축 URL + QR코드** 제공, 클릭수 추적 |
| 메인 메뉴 | **모든 메뉴를 관리자가 표시명·링크주소·순서·노출·그룹까지 변경** 가능(예외 없음, `menu_items` 데이터 관리) |
| 제품 양식 기준 | KOTRA **buyKOREA 제품상세 구조** 벤치마크(인증/수상·MOQ·리드타임·운송·결제·출고지·카탈로그 등) |
| 법적 페이지 | 이용약관·개인정보처리방침·쿠키정책(en/ko). **초안은 placeholder, 오픈 전 변호사 검토 필수** |
| 회원 탈퇴 | 기본 **익명화 보존**(이력 무결성), 삭제권 요구 시 개인정보 파기 + 감사기록 |
| 쿠키 동의 | 필수 쿠키만 예외. 비필수(분석·마케팅·기능)는 **opt-in 전 차단**. 수락=거부 동등 배치, 철회 가능, 동의기록 보관 |
| UX 안내 | 주요 행동은 **확인→처리중→결과** 3단계. 파괴적 행동은 확인 필수, 성공·실패 항상 피드백(8장) |
| 비밀번호 | 표시/숨김 토글(눈 아이콘) 기본, 강도·규칙 인라인 안내(8.2) |
| 로딩 | 버튼 스피너·스켈레톤·진행률로 "멈춤" 오인 방지, 빈 상태·오류 상태 안내(8.3) |
| 검색 | 필요한 화면은 **실시간 검색(as-you-type)**, 디바운스·스켈레톤·결과없음 안내, 권한/게이트 준수(8.5) |

## 12. 의도적 보류 (MVP 이후)

1. **에이전트 보상 정산 규칙** — 재원(누가 대는가)·기록·분쟁처리 미정. 실제 거래가 돌기 시작한 뒤 설계. 정해지면 에이전트에게 **거래금액 공개 여부**도 함께 재검토.
2. **제품 등록 세부 항목** — MOQ, 리드타임, 수출절차, 포장단위, 결제조건, 인증 등은 **공급사 입력 필드**로 정의. 일부 고정 필드 + 일부 자유 추가 방식으로 확장.
3. **좋은 에이전트 선별의 명문 기준** — 현재는 관리자 재량 선택제로 운영.
4. **추가 언어 확장** — 영어·한국어 외 언어는 특정 국가 바이어 수요 확인 후.

### 12.1 점검에서 확인된 열린 항목 (구현 전 결정 필요)
- **신고·차단(트러스트&세이프티)** — 회원제·검증 플랫폼이라 우선순위는 낮으나, 부적절 제품/요청글 신고 경로가 아직 없음. 필요 시 `reports` 테이블로 추가.
- **관리자 역할 세분화** — 현재 role enum의 admin은 단일. 운영자가 여럿이 되면 슈퍼관리자/일반운영자 구분과 권한 범위가 필요할 수 있음(감사로그와 연계).
- **로그인 회원의 랜딩 재방문 동선** — 로그인 시 대시보드로 진입하므로, 활기찬 제품 랜딩·목록으로 돌아오는 내비게이션(로고/"제품 둘러보기")을 구현 시 확정.

### 12.2 UI · 디자인 시스템 (미확정)
- 시각 디자인 방향(색·타이포·컴포넌트 토큰)은 **아직 확정 전**이다. 별도 도구(예: v0)로 시안을 탐색 중이며, 방향이 정해지면 색 팔레트·타이포 스케일·간격·컴포넌트 규칙을 이 문서에 디자인 토큰 섹션으로 추가한다. 단, 6장의 랜딩 **구성·콘텐츠·열람 경계**는 시각 스킨과 무관하게 유효하다.

## 13. 개발 로드맵 (권장 순서)

**Phase 1 — 기반**
1. Supabase 프로젝트 · Next.js · GitHub 연결, 배포 파이프라인
2. i18n 언어팩 기반 구성(영어 기본 + 한국어), 환경변수·상수 모듈 세팅 (0.1 규칙 준수)
3. `profiles` + Auth(이메일), 역할 4종, RLS 뼈대
4. 초대/추천 코드 가입 흐름(A)
5. 법적 페이지(약관·개인정보·쿠키정책 placeholder) + **쿠키 동의 배너**(opt-in, 수락=거부 동등)
6. **이메일 발송 기반**(트랜잭션 메일 서비스 연동, `email_outbox`) — 가입 인증·비밀번호 재설정부터
7. **공통 UX 컴포넌트**(확인 다이얼로그·토스트·스켈레톤·비밀번호 토글·빈/오류 상태) — 8장 표준을 전 화면에서 재사용

**Phase 2 — 진열대**
7. 공급사 가입·승인, 제품 등록(buyKOREA 구조 참고: 이미지·인증/수상·MOQ·리드타임·운송·결제 등), 관리자 노출 승인(B) + **승인/반려 이메일 발송**
8. 목록/상세/가격 회원제 접근 제어, 공급사 미니홈페이지, 대분류 카테고리, **제품 실시간 검색·필터**(8.5)

**Phase 3 — 심장(중개)**
9. 문의·견적 생성 → 관리자 중개 → 공급사 익명 전달 → 회신(C) + **문의 도착·회신 이메일 발송**(공급사향은 신원 가림)
10. 역할별 대시보드 4종, 알림(앱 내 토스트/알림박스 + 이메일 연계)
11. **관리자 회원 전권 관리**(목록·상세·수정·역할/상태 변경·감사로그)

**Phase 4 — 게시판 · 성장**
12. 메인 메뉴/서비스 관리자 편집(`menu_items`, `services`), EPC 프로젝트(`projects`)
13. **공통 게시판 에디터·첨부·인라인 미디어**(`board_attachments`, 이미지·동영상 인라인 재생/임베드) + 관리자 공지형 게시판: 공지사항·FAQ·행사(참여상태 포함) + 행사 리마인더 메일
14. Sourcing Requests(RFQ): 작성 → 관리자 승인 노출 → 공급사 응답 → 관리자 중개
15. **공유 단축 URL + QR코드**(제품 공유·추천가입), 클릭 추적
16. 유료/무료 등급, 인증마크, 추천제품 편성, **랜딩 광고배너·팝업 관리**
17. 익명 활동 신호, 에이전트 하부 바이어 관리
18. (이후) 정산·추가 언어·세부 제품필드·Networking 확장
